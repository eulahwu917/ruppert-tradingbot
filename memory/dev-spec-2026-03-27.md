# Dev Spec — 2026-03-27 Evening Batch

## Bug 1: Settlement Checker + pnl_cache Update (Combined)

### Problem
In DEMO mode, trades are never actually placed on Kalshi, so the API never reports settlement outcomes. The bot has no way to know if it won or lost. Additionally, when `post_trade_monitor.py` executes an auto-exit, it logs the trade but never updates `pnl_cache.json`. Capital tracking always shows $0 realized P&L.

### Solution

#### Part A: Settlement Checker in `post_trade_monitor.py`

Add a new function `check_settlements()` that runs each cycle alongside the existing position monitor.

Logic:
1. Load all open DEMO positions from today's + yesterday's trade log (buys not yet exited)
2. For each position where `target_date < today` OR `close_time` has passed:
   - Fetch the market from Kalshi API via `kalshi_client.get_market(ticker)`
   - Check if market is resolved: look for `result` field (`yes`/`no`) OR `status == 'finalized'` OR `yes_bid >= 99` / `yes_bid <= 1`
   - If resolved, compute simulated P&L:
     - If side=`yes` and result=`yes`: `pnl = (99 - entry_price) * contracts / 100`
     - If side=`yes` and result=`no`: `pnl = -(entry_price * contracts / 100)`
     - If side=`no` and result=`no`: `pnl = (99 - entry_price) * contracts / 100`
     - If side=`no` and result=`yes`: `pnl = -(entry_price * contracts / 100)`
     - Where `entry_price` = `fill_price` from trade log (fallback to `scan_price`, fallback to `market_prob * 100`)
   - Log a `action=settle` entry to today's trade log with ALL of these fields:
     ```json
     {
       "trade_id": "<uuid>",
       "timestamp": "<now>",
       "date": "<today>",
       "ticker": "<ticker>",
       "module": "<weather|crypto|etc>",
       "side": "<yes|no>",
       "action": "settle",
       "settlement_result": "<yes|no>",
       "entry_price": <float>,
       "exit_price": <99 or 1>,
       "contracts": <int>,
       "pnl": <float>,
       "size_dollars": <entry cost>,
       "entry_edge": <float>,
       "confidence": <float>,
       "hold_duration_hours": <float>,
       "source": "settlement_checker",
       "dry_run": true
     }
     ```
   - Update `pnl_cache.json` by incrementing `closed_pnl` by the pnl amount
   - Add ticker to `traded_tickers` so it's not re-processed

Call `check_settlements()` at the TOP of `post_trade_monitor.py`'s main loop, before position checks.

#### Part B: pnl_cache Update on Auto-Exit

In `post_trade_monitor.py`, when an auto-exit fires (95c rule or 70% gain rule), update `pnl_cache.json`:
- Read current `closed_pnl` from `pnl_cache.json`
- Add the exit P&L
- Write back

`pnl_cache.json` format (already exists):
```json
{"closed_pnl": 0.0}
```

Use a file lock pattern (read → update → write) to avoid race conditions with multiple monitor instances.

### Notes
- All of this runs in DEMO (dry_run=True) — never place real orders
- entry_price: use `fill_price` from original trade log entry; if null use `scan_price`; if null use `market_prob * 100`
- If Kalshi API returns no result yet (market still open), skip silently
- Log errors non-fatally — never crash the monitor

## Bug 2: Daily Progress Report Telegram Truncation

### Problem
`daily_progress_report.py` sends via `send_telegram()` in `logger.py`, which calls `openclaw message send -m <message>` as a subprocess with `shell=True`. Multiline messages are being truncated — only the first line arrives in Telegram.

### Solution
In `logger.py`, the `send_telegram()` function currently passes the message as a command-line argument:
```python
result = subprocess.run(
    ['openclaw', 'message', 'send', '--channel', 'telegram', '-t', '5003590611', '-m', message],
    capture_output=True, text=True, timeout=30, shell=True,
)
```

The issue is that `shell=True` with a list on Windows collapses arguments differently, and long multiline strings get mangled. Fix by writing the message to a temp file and using stdin, OR switch to passing via `--file` if openclaw supports it, OR use `shell=False` with explicit quoting.

**Preferred fix:** Remove `shell=True` and ensure the message is passed correctly:
```python
result = subprocess.run(
    ['openclaw', 'message', 'send', '--channel', 'telegram', '-t', '5003590611', '-m', message],
    capture_output=True, text=True, timeout=30,
    # shell=True removed
)
```

Test by running `daily_progress_report.py` manually and verifying full message arrives in Telegram with all sections intact.

## Bug 3: Crypto scan_price / fill_price Null Fields

### Problem
Crypto trade log entries show `scan_contracts: null, fill_contracts: null, scan_price: null, fill_price: null`. Weather trades log these correctly. This affects exit calculations in `run_exit_scan()` which uses these fields to determine entry price — when null it falls back to `market_prob * 100` (approximation).

### Root cause
In `main.py`, `run_crypto_scan()` builds the `opp` dict for logging (around line 645) but doesn't populate `scan_price`, `fill_price`, `scan_contracts`, `fill_contracts`. These are set in the trade log from the `opp` dict fields.

### Fix
In `run_crypto_scan()`, when building the opp dict for `log_trade()`, add:
```python
'scan_price': t['price'],
'fill_price': t['price'],  # DEMO: no real fill, use scan price
'scan_contracts': contracts,
'fill_contracts': contracts,
```
Where `t['price']` is the yes_ask or no_ask price used for sizing, and `contracts` is the computed contract count.
