## Bug 1: Chicago — mutually exclusive weather positions

### Problem
`find_opportunities()` can return multiple opportunities for the same city + settlement date (e.g., Chicago Mar 27: T44, T37, B41.5, B39.5 all passing the edge threshold). These represent mutually exclusive temperature outcomes — betting multiple contradictory positions simultaneously is logically invalid and wastes capital.

### Fix Applied
After `opportunities = find_opportunities(markets)` in `run_weather_scan()`, added a dedup block:

```python
# ── Dedup: keep only highest-edge opp per (city, settlement_date) pair ──
opportunities = sorted(opportunities, key=lambda o: o.get('edge', 0), reverse=True)
_seen_city_date = {}
_deduped_opps = []
for opp in opportunities:
    _city = opp.get('city') or opp.get('ticker', '').split('-')[0]
    _date = opp.get('target_date')
    _key = (_city, _date)
    if _key in _seen_city_date:
        _winner_ticker = _seen_city_date[_key]
        log_activity(f"[Weather] DEDUP skip {opp.get('ticker')} — city={_city} date={_date} already covered by {_winner_ticker}")
    else:
        _seen_city_date[_key] = opp.get('ticker')
        _deduped_opps.append(opp)
opportunities = _deduped_opps
```

### Reasoning
- Sort descending by edge first so the best opportunity wins the dedup
- Key is `(city, target_date)` — if `city` is None, falls back to series prefix from ticker
- Logs each skipped opp for auditability
- Inserted before the "Found N opportunities" log so the logged count reflects the deduped list

## Bug 2: Crypto confidence > 1.0

### Problem
In `run_crypto_scan()`, `_spread_score` was computed as `max(0.0, 1.0 - (_spread / 20.0))` with only a floor clamp. When `no_ask > yes_ask` (inverted spread / negative `_spread`), the value exceeds 1.0, which then pushes the composite `_crypto_confidence` above 1.0.

### Fix Applied
**Line ~552** in `run_crypto_scan()`:

Before:
```python
_spread_score = max(0.0, 1.0 - (_spread / 20.0))
...
_crypto_confidence = round((_edge_score * 0.5 + _spread_score * 0.3 + _time_score * 0.2), 3)
```

After:
```python
_spread_score = min(1.0, max(0.0, 1.0 - (_spread / 20.0)))
...
_crypto_confidence = round(min(1.0, max(0.0, _edge_score * 0.5 + _spread_score * 0.3 + _time_score * 0.2)), 3)
```

### Reasoning
- `_spread_score` now has both floor (0.0) and ceiling (1.0) clamps
- Final composite also clamped [0.0, 1.0] as a defense-in-depth measure
- No semantic change for normal spreads; only affects inverted-spread edge cases
- `_edge_score` and `_time_score` already had `min(1.0, ...)` — this brings `_spread_score` to parity
