## ⚠️ Warnings (Discretionary)

### W1 — `api.py` ~L199 + L207–211: Duplicate dict keys in `get_account()` return
Three keys appear twice in the return dict:
- `"starting_capital"` at L199 (unrounded) and L207 (rounded)
- `"bot_deployed"` at L208 and L210
- `"manual_deployed"` at L209 and L211

Python silently uses the **last** value, so the rounded versions win — functionally correct. But this is confusing and a linting error. Should be cleaned up in the next pass.

### W2 — `api.py` `is_settled_ticker()`: Hardcoded UTC-4 (EDT), no DST awareness
```python
now_hour_edt = (_dt.utcnow().hour - 4) % 24
```
Assumes EDT (UTC-4) year-round. When clocks fall back to EST (UTC-5) in November, hourly markets on the same day could be misjudged settled by 1 hour. Low probability of impact given demo phase, but worth noting before live trading.

### W3 — `api.py` `set_mode()`: No error handling on write
```python
def set_mode(mode: str):
    MODE_FILE.write_text(json.dumps({"mode": mode}, indent=2), encoding='utf-8')
```
If the disk write fails (permissions, disk full), the endpoint still returns `{"ok": True}`. Silent false-positive acknowledgment. Mode would revert to `demo` on next read. Add try/except and return error on failure.

### W4 — `api.py` `execute_highconviction()`: `is_live` hardcoded False
```python
is_live = False  # TODO: read from config when going live
```
This endpoint always runs in demo mode regardless of `get_mode()`. When David switches to live, this endpoint will still log demo trades instead of placing real orders. Must be wired to `get_mode()` before live launch.

### W5 — `index.html` `switchMode()`: UI updates even on API failure
```python
async function switchMode(mode) {
  try {
    await api('/api/mode', {...});
    applyMode(mode);   // ← called even if api() returned null
    loadAccount();
  } catch(e) { ... }
}
```
`api()` swallows its own exceptions and returns `null`. The outer `try/catch` will not catch an API failure. If the POST fails, the toggle still visually flips to LIVE. Corrects itself on next `loadMode()` cycle (~30s), but there's a brief inconsistency window. Low risk in practice; worth hardening before live trading.

### W6 — `index.html` `loadLivePrices()`: `window._openPnl` can be stale when positions go to zero
When `loadPositions()` finds no open positions, it returns early without calling `loadLivePrices()`. If positions previously existed, `window._openPnl` retains its last value. `updateAccountValue()` would show a non-zero open P&L until the next `loadAll()` cycle.

The `if (true)` change correctly solves the `totCost=0` case (positions exist but have zero cost) — that specific fix is good. The zero-positions stale-state issue is a **pre-existing bug** not introduced today.

### W7 — `index.html` `loadCrypto()`, `loadGeo()`, `loadWeather()`: Referencing DOM elements that don't exist
Functions reference `$('crypto-b')`, `$('crypto-sm-badge')`, `$('crypto-btc-price')`, `$('crypto-btc-chg')`, `$('crypto-eth-price')`, `$('crypto-eth-chg')`, `$('geo-list')`, `$('w-b')` — none present in the current HTML template. These are silent null-pointer failures (no throw, just `null.textContent` or `null.innerHTML` errors suppressed by the tab not calling them). Pre-existing issue; not introduced today.
