**Code path is identical by construction.**

- `/api/pnl` modules: iterates `get_trades()` and sums `realized_pnl` per module

Same function → same data source → same `realized_pnl` values → sum of module P&Ls will mathematically equal sum of `realized_pnl` from `/api/trades`. No intermediate transformation or filtering divergence exists between the two paths.

`get_trades()` already excludes manual trades (`source in ('manual', 'economics', 'geo')`), so both the module cards and the closed table consistently exclude manual trades.

## ⚠️ WARNING: Date Bucketing Inconsistency (Pre-Existing)

**Not introduced by this fix — inherited from prior design.**

Module-level `closed_pnl_month` / `closed_pnl_year` use the **trade entry date** (`_ct.get('_date')` from the trade log filename):

```python
_date_str = (_ct.get('_date') or _ct.get('date', ''))[:10]
```

Account-level `closed_by_period['month']` uses the **settlement date** (parsed from ticker) or exit timestamp:

```python
if ticker in exit_records and exit_records[ticker].get('timestamp'):
    sdate = datetime.fromisoformat(exit_records[ticker]['timestamp']).date()
else:
    sdate = settlement_date_from_ticker(ticker)
```

**Impact:** A trade entered in February that settles in March will appear in February's module monthly total but March's account monthly total. This is a discrepancy between module-level and account-level monthly P&L — though given typical Kalshi market durations (1–7 days), practical impact is likely minimal.

**Recommendation:** Low priority. Consider aligning module date bucketing to use settlement/exit date for future consistency, but this is pre-existing behavior and not blocking.
