`main.py` has NO function named `scan_weather`. The function is `run_weather_scan`. This throws an `ImportError` every time `ruppert_cycle.py` runs in `full` mode (7am, 3pm cycles). Step 3 (weather scan) is completely broken.

### 2. `ruppert_cycle.py` — TypeError in weather position check (Step 1)
**Line ~95:** `sig = get_full_weather_signal(city)`  
`get_full_weather_signal` requires **3 positional arguments**: `(series_ticker: str, threshold_f: float, target_date: date)`. Only 1 is passed.  
Additionally, `city` comes from `city_map` (e.g. `'miami'`, `'new_york'`) but the function expects a series ticker like `'KXHIGHMIA'`. This is a **double error**:  
- Wrong number of arguments → always throws `TypeError`  
- Wrong key format even if arity were fixed  
Caught by `except Exception as e: print(...)` so it fails silently. Weather-based position alerts and auto-exits **never execute**.  
**Fix:** Pass correct args: `get_full_weather_signal(series_ticker, threshold_f, target_date)`, where `series_ticker` is derived from the ticker (e.g. `'KXHIGHMIA'`).

### 3. `dashboard/api.py` — NameError on HC approve and pass endpoints
**`approve_highconviction` and `pass_highconviction` endpoints:**  
```python
log_path = BASE_DIR / 'logs' / 'highconviction_approved.jsonl'
log_path = BASE_DIR / 'logs' / 'highconviction_passed.jsonl'
```

### 4. `edge_detector.py`, `main.py`, `ruppert_cycle.py` — Wrong strptime format (always fails silently)
**Locations:**
- `edge_detector.py` line ~54: `datetime.strptime("20" + date_str, "%Y%d%b%d")`
- `main.py` `run_exit_scan()` line ~131: `datetime.strptime("20" + date_part, "%Y%d%b%d")`
- `ruppert_cycle.py` position check line ~60: `datetime.strptime("20" + date_part, "%Y%d%b%d")`

The format `"%Y%d%b%d"` is wrong. For input `"2026MAR11"`:  
- `%Y` consumes `"2026"` (year = 2026)  
- `%d` tries to match next chars `"MAR11"` — `"M"` is not a digit → `ValueError` every call  
The correct format is `"%Y%b%d"`.

**Consequences:**
- `edge_detector.py`: `parse_date_from_ticker()` always falls back to `date.today()`. Opportunities for tomorrow's markets get `target_date = today`, which compresses `hours_to_settlement` and can falsely trigger the "too close to settlement" gate, **silently blocking valid entries**.
- `main.py` exit scan: falls back to `hours_to_settlement = 24.0` (safe but wrong — near-settlement hold never fires correctly).
- `ruppert_cycle.py` Step 1: falls back to `24.0` hours (safe fallback, but the near-settlement logic is never exercised).

**Fix:** Change `"%Y%d%b%d"` → `"%Y%b%d"` in all three files.

### 5. `ruppert_cycle.py` crypto loop — No daily cap enforcement
**Lines ~145–230 (Step 4, crypto scan):**  
The crypto loop executes up to 3 trades per run with no check against the 70% daily deployment cap. It uses `size = min(config.CRYPTO_MAX_POSITION_SIZE, 25)` = $25 unconditionally and calls `log_trade()` directly, bypassing `strategy.py`'s `check_daily_cap()` and `should_enter()`.  
If the weather scan already deployed near the daily cap, crypto trades would still execute, potentially breaching the 30% reserve rule.  
**Fix:** Add a daily cap check before executing crypto trades in `ruppert_cycle.py`, using `check_daily_cap(total_capital, deployed_today)` from `bot/strategy.py`.

### 6. `logger.py` — Missing `encoding='utf-8'` on all file opens
All 5 file open calls in `logger.py` are missing `encoding='utf-8'`:
- `log_trade()` line ~33: `with open(_today_log_path(), 'a') as f:`
- `log_opportunity()` line ~45: `with open(_activity_log_path(), 'a') as f:`
- `log_activity()` line ~52: `with open(_activity_log_path(), 'a') as f:`
- `get_daily_exposure()` line ~60: `with open(log_path, 'r') as f:`
- `get_daily_summary()` line ~72: `with open(log_path, 'r') as f:`

On Windows, this uses the system default encoding (often cp1252), which will fail on any trade ticker or title containing non-ASCII characters, and may corrupt JSONL log files.  
**Fix:** Add `encoding='utf-8'` to all `open()` calls in `logger.py`.

### 9. `index.html` — JavaScript `api()` ignores fetch options; mode toggle broken
**`switchMode()` function:**
```javascript
async function api(url) { try { return await (await fetch(url)).json(); } catch { return null; } }
...
await api('/api/mode', { method: 'POST', headers: {...}, body: JSON.stringify({mode}) });
```
The `api()` helper only accepts `url` — the second argument (fetch options) is silently ignored. This means `switchMode()` sends a **GET** request to `/api/mode` instead of POST, returning the current mode without changing it. **The mode toggle in the dashboard does not work.**  
**Fix:** Either extend `api(url, options={})` to pass options to `fetch()`, or use `fetch()` directly in `switchMode()`.

## ⚠️ Warnings (Discretionary)

**W1. `dashboard/api.py` `get_account()` — Duplicate dict keys**  
`"starting_capital"`, `"bot_deployed"`, and `"manual_deployed"` each appear twice in the returned dict literal. Python silently uses the last value for duplicate keys. Currently the last values are the correct rounded ones, so functionally OK, but this is fragile and should be cleaned up.

**W2. `ruppert_cycle.py` — Multiple bare `except: pass` blocks**  
`push_alert()` line ~54, and the open_positions parsing loop use `except: pass`. These silently swallow errors including JSON parse failures and OS errors. Should use `except Exception as e: print(f'...: {e}')` at minimum.

**W3. `logger.py` — Bare `except: pass` in `get_daily_exposure()` and `get_daily_summary()`**  
If the trade log is malformed, `get_daily_exposure()` returns 0.0 silently, which would make the daily cap appear unlimited and allow over-deployment. Should log the error.

**W4. `dashboard/api.py` — Multiple bare `except: pass` blocks**  
In `read_all_trades()`, `read_high_conviction()`, P&L calculation loops, and price fetches. Silent failures make debugging difficult. Should at minimum log to stderr.

**W5. `crypto_scanner.py` vs `dashboard/api.py` — Cache file name mismatch**  
`crypto_scanner.py` writes scanner results to `logs/crypto_scan_latest.json` (JSON format).  
`dashboard/api.py` `get_crypto_scan()` looks for `logs/crypto_scan.jsonl` (JSONL format).  
The cache is never found → dashboard falls back to raw market data with no edge analysis.

**W6. `crypto_client.py` — 5 of 8 Polymarket wallets are placeholders**  
`TOP_TRADER_WALLETS` contains `'0xTODO_wallet_placeholder_4'` through `'...8'`. The smart money signal is based on 3 real wallets + 5 that return HTTP errors. The signal is unreliable until real wallet addresses are added (noted as TODO in team_context.md).

**W7. `kalshi_client.py` — DEMO_HOST equals PROD_HOST**  
`DEMO_HOST` and `PROD_HOST` are set to the same URL. When switching to live trading, there is no URL differentiation. The host selection logic (`DEMO_HOST if environment == 'demo' else PROD_HOST`) is a no-op. Acceptable now (demo account uses same API with different credentials) but fragile for future environment management.

**W8. `dashboard/api.py` `get_summary()` — Hardcoded `"DRY RUN"` mode**  
```python
return {"mode": "DRY RUN", "status": "RUNNING", ...}
```
Ignores the actual mode from `mode.json`. Returns `"DRY RUN"` even in live mode.

**W9. `bot/strategy.py` not yet wired into `main.py` for add-on decisions**  
`run_weather_scan()` calls `should_enter()` for every opportunity each cycle, but there is no mechanism to call `should_add()` for tickers already in the portfolio. If the same ticker surfaces in two consecutive scan cycles, `should_enter()` would approve a second $25 entry, potentially breaching the $50 max-per-ticker cap. Noted as a known pending item in `team_context.md`.

**W10. `dashboard/api.py` `get_pnl_history()` — Hardcoded chart date**  
```python
points.append({"date": "2026-03-10", "pnl": round(closed_pnl_total, 2)})
```
The historical P&L chart always anchors at 2026-03-10, regardless of the actual first trade date. The chart will become misleading as time passes.

**W11. `position_monitor.py` `check_near_settlement()` — DST offset off by 1 hour**  
The code uses a fixed UTC-5 offset for Eastern time, but March is EDT (UTC-4). The settlement time calculation will be 1 hour off during EDT season (March–November), potentially letting contracts within 30–90 minutes of settlement bypass the hold rule.

**W12. `position_monitor.py` `load_open_positions()` — Uses latest entry per ticker, not earliest**  
`by_ticker[ticker] = entry` keeps the last-seen trade record. If a ticker appears in multiple log files (e.g., an add-on), the latest record overwrites the original entry price. P&L calculations use `market_prob` from the latest record, not the original entry. Should track the first entry per ticker for P&L accuracy.

**W13. `logger.py` `log_trade()` — No explicit `entry_price` field logged**  
Exit P&L is computed by deriving entry price from `market_prob`, which reflects the market probability at scan time, not the actual fill price. In fast markets this can differ. Suggest adding `entry_price_cents` explicitly to the trade log record.

**W14. `bot/strategy.py` `should_enter()` — `deployed_today` not refreshed between trades in same cycle**  
In `run_weather_scan()`, `deployed_today` is fetched once before the loop. If 3 trades execute in one scan, all three use the same pre-loop `deployed_today` value. The 3rd trade sees the same remaining cap as the 1st. At current capital ($400, $25 trades) this is unlikely to cause a cap breach, but should be fixed before scaling.
